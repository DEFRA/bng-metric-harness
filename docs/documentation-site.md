## Documentation site

A unified documentation site published from this repo, aggregating content across the three BNG Metric repos.

**Live at: <https://defra.github.io/bng-metric-harness/>**

## What gets published

- One section per repo (`/harness/`, `/frontend/`, `/backend/`), each containing that repo's `README.md` plus everything in its `docs/` folder, rendered as separate pages with a left-hand index.
- An **Architecture** section embedding an interactive [LikeC4](https://likec4.dev/) diagram explorer built from the `.likec4` sources in this repo. Also available standalone at `/architecture/app/`.
- A **Build info** page showing the HEAD SHA of each sibling repo at the time of the most recent build.

Markdown is rendered with MkDocs Material — GFM tables become real tables, fenced code blocks are syntax-highlighted, and there's a search box across all pages.

## How it rebuilds

The site is rebuilt by `.github/workflows/pages.yml` in two scenarios:

- **Nightly** at 05:00 UTC. This is how documentation changes in sibling repos make it onto the published site without anything else having to happen.
- **On manual trigger** via the Actions tab — useful after you push a doc change and don't want to wait until the next morning.

There's no auto-publish on push. Doc changes land in `main` (of the harness or a sibling) and then either wait for the cron or get pulled forward by a manual trigger.

## Triggering a manual publish

From the GitHub UI:

1. Open the [Build & deploy GitHub Pages workflow](https://github.com/DEFRA/bng-metric-harness/actions/workflows/pages.yml).
2. Click **Run workflow** → **Run workflow** (on `main`).
3. Watch the `build` then `deploy` job. The `deploy` job's summary surfaces the live URL once it succeeds.

Or from the CLI:

```sh
gh workflow run pages.yml --ref main
gh run watch
```

The whole cycle (checkout three repos → aggregate → build LikeC4 SPA → build MkDocs site → upload artifact → deploy) takes ~2–3 minutes.

## Adding a new documentation page

Drop a `.md` file into the `docs/` folder of whichever repo it belongs to (`bng-metric-harness/docs/`, `bng-metric-frontend/docs/`, or `bng-metric-backend/docs/`). The next nightly build (or manual trigger) picks it up automatically — the aggregator walks each sibling's `docs/` directory and writes everything into the published site, no per-file registration anywhere.

Conventions:

- Use a lower-case-hyphenated filename — it shows up in the URL verbatim. The page title in the left-hand nav is auto-generated by title-casing the filename (so `database-changes.md` becomes "Database Changes"; common acronyms like CSRF / API / SQL are upper-cased).
- Use GFM tables, code fences, admonitions — all supported.
- Links between docs in the same `docs/` folder work as relative paths (`[CSRF](csrf.md)`). Links outside `docs/` (`LICENSE`, `src/foo.js`, etc.) are rewritten to point at the canonical file on GitHub at `HEAD`.

## Building locally

### Dev Container (recommended)

See the [Dev Container section in the README](../README.md#local-development-dev-container). After **Reopen in Container**, MkDocs is pre-installed and **MkDocs: serve** can start on folder open.

### Manual setup

```sh
# Node deps (already installed if you ran `npm install`)
npm install

# MkDocs Material (Python). Pipx keeps it out of your global env.
pipx install mkdocs-material

# Aggregate + build LikeC4 + serve with live reload
npm run docs:serve
# → http://localhost:8000/bng-metric-harness/
```

Useful npm scripts:

| Command | What it does |
| ------- | ------------ |
| `npm run docs:aggregate` | Copies markdown from sibling repos and emits `mkdocs.generated.yml`. |
| `npm run docs:likec4` | Builds the LikeC4 static SPA into `site_src/docs/architecture/app/`. |
| `npm run docs:build` | All of the above + `mkdocs build` → `_site/`. |
| `npm run docs:serve` | All of the above + `mkdocs serve` (live reload). |

## One-time setup

For the workflow to be able to deploy, GitHub Pages must be enabled with Actions as the source:

1. Open <https://github.com/DEFRA/bng-metric-harness/settings/pages>.
2. Under **Build and deployment**, set **Source** to **GitHub Actions**.

If this isn't done, the workflow's `Setup Pages` step fails with a clear error rather than producing a broken deploy.

## Troubleshooting

- **Workflow failed at `Setup Pages`** — Pages isn't enabled for the repo. See the one-time setup above.
- **Workflow failed at `mkdocs build --strict`** — a markdown file has a broken internal link or a structural issue. The job logs show the exact file and line.
- **Architecture iframe shows a 404** — the LikeC4 build step didn't run, or its output didn't land under `site_src/docs/architecture/app/`. Check the workflow logs.
- **Sibling doc not appearing** — was the file added under that sibling's `docs/` folder (not e.g. `documentation/` or `doc/`)? The aggregator only walks `docs/`.
